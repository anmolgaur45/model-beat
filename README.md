# Model Beat

Covering the AI beat, every day. Model Beat is a date-organized timeline for AI news. Stories are pulled from dozens of sources, deduplicated with sentence embeddings, and ranked by a significance score that weighs source authority and how many outlets covered the same story. It also tracks the models themselves: a registry of recent releases with benchmark scores, pricing, and the news coverage of each launch.

Live at [themodelbeat.com](https://themodelbeat.com).

## Architecture

```
Vercel (Frontend)                      Google Cloud (Backend)
┌──────────────────────┐     ┌────────────────────────────────────┐
│ Next.js + tRPC       │ SSL │ Cloud SQL (Postgres + pgvector)     │
│ Timeline + /models   │────▶│                                     │
│ /api/revalidate      │◀────┼─ Cloud Run Job (Python)             │
└──────────────────────┘ ISR │   ingest → embed → score → cluster  │
                             │   → merge → summarize → sync models  │
                             │                                     │
                             │ Cloud Scheduler (every 3h)          │
                             └────────────────────────────────────┘

External data: Epoch AI (benchmarks), OpenRouter (pricing and specs),
Gemini on Vertex AI (cluster summaries), Claude Haiku (impact scoring).

GitHub Actions: on a push to pipeline/, run the tests, build the
image, and deploy the Cloud Run Job (keyless Workload Identity auth).
```

The frontend reads from Cloud SQL over SSL. When the pipeline finishes a run it calls `/api/revalidate` so the frontend flushes its cache and new stories show up without a redeploy.

## Features

### News timeline
- Timeline view, with stories grouped by date and ranked by significance within each day
- Catch me up, a recap of the highest-scoring stories from the last seven days grouped by day, with a minimum score floor so it only surfaces what actually mattered
- Deduplication with `all-MiniLM-L6-v2` embeddings, so the same story from several outlets becomes a single card with every source listed
- Significance scoring from source authority, the spread of distinct outlets covering a story, and a per-article impact score
- AI summaries written per cluster, generated by Gemini on Vertex AI
- Quiet-day signalling, where a day on which nothing cleared the notable threshold is labeled as such instead of being padded out
- Hacker News discussion links on any story that was also covered on HN
- Category filters: Models, Research, Companies, Products, Policy, Hardware, Open Source, Opinion
- Full-text search with PostgreSQL FTS and synonym expansion
- Every card cites its original source with a timestamp

### Model tracker
- A registry at `/models` of notable models released over the past year, with benchmark scores from Epoch AI and pricing and specs from OpenRouter
- A leaderboard that re-ranks by use case (Overall, Coding, Math, Reasoning, Agentic), where each ranking is a percentile composite over the benchmarks relevant to that area
- A side-by-side compare view at `/models/compare` for two to four models
- Charts that plot price against quality and the moving frontier over time
- A page for each model with its benchmark scores, specs, pricing, and the news coverage of its release

### SEO and syndication
- Per-day archive pages, an XML sitemap, and a `robots.txt`
- Structured data (JSON-LD) for the site, articles, and model pages
- An RSS feed at `/feed.xml`
- A free weekly email digest, with signup at [/digest](https://themodelbeat.com/digest)

## Running locally

### Frontend

```bash
cd frontend
pnpm install
cp .env.local.example .env.local  # fill in DB credentials
pnpm dev
```

Open [http://localhost:3000](http://localhost:3000). To open the app from a phone on the same network, add your machine's LAN IP to `allowedDevOrigins` in `next.config.ts` and use the `Network` URL that `pnpm dev` prints.

### Pipeline

```bash
cd pipeline
python -m venv .venv
source .venv/bin/activate   # or .venv\Scripts\activate on Windows
```

## Database

Migrations live in `pipeline/migrations/` and target PostgreSQL 15 with the `vector` extension. The schema includes:

- `articles`, ingested articles with 384-dimensional embeddings and impact scores
- `clusters`, deduplicated story groups with significance scores, AI summaries, and an FTS index
- `models`, `model_benchmarks`, and `model_clusters`, the model registry, its benchmark scores, and the links between a model and the stories that covered it
- `model_events`, detected changes to a model's pricing or context window, diffed from the OpenRouter catalog on every run
- `pipeline_runs`, a log of run timestamps used to flag stale data in the UI
- `find_nearest_article()`, the pgvector ANN search function used during clustering

## Deduplication and scoring

Articles are embedded with `all-MiniLM-L6-v2`, which produces 384-dimensional vectors. For each new article, an approximate-nearest-neighbor search looks for similar articles within 48 hours of its publication date. An article joins an existing cluster only when it sits close to both a member article and the cluster centroid, which keeps unrelated stories from chaining together. Thresholds are set per source: news articles cluster at a cosine distance of 0.30, while arXiv papers use a tighter 0.10 so that distinct papers on the same topic stay apart.

Clusters are built one article at a time, so a story whose coverage is spread out can occasionally split across two clusters. A merge pass at the end of every run handles that case. It compares cluster centroids inside the time window and merges any pair closer than the threshold, so a single story never shows up as two cards.

The significance score weighs the authority of the sources covering a story. A base score adds up the covering organizations with diminishing returns, so a few authoritative outlets count for more than a long tail of low-authority ones. A coverage multiplier scales the same way, weighing each distinct organization by its authority rather than counting heads, so a wire story syndicated across many small sites cannot outrank first-hand reporting from a major outlet. That base is multiplied by the highest per-article impact score in the cluster, and the raw value is compressed onto a 1 to 10 scale with a logarithmic curve so the biggest stories keep some separation instead of flattening out at the maximum.

## Deployment

The frontend deploys to Vercel automatically on every push to `main`, and serves from [themodelbeat.com](https://themodelbeat.com).

The pipeline deploys through GitHub Actions. A push that touches `pipeline/` runs the test suite first, and on success builds the container image and deploys it to the Cloud Run Job, pinned to the commit SHA. Authentication uses Workload Identity Federation, so there are no service-account keys in the repository. First-time setup for the federation pool, the deployer service account, and the IAM roles is documented in `pipeline/DEPLOY.md`.

Cloud Scheduler triggers the pipeline every three hours. When a run completes it calls `/api/revalidate` on the frontend to refresh the cache.

## Observability and security

- Page and performance metrics through Vercel Analytics and Speed Insights
- Error tracking on the client, server, and edge runtimes
- A `pipeline_runs` log of every run, with email alerts on failed Cloud Run executions
- Security headers on every response, including Content-Security-Policy, X-Frame-Options, and (in production) HSTS
- In-memory rate limiting on the write endpoints
- All API inputs validated with Zod, including allowlists for category and date parameters
